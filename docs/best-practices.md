# Soak Testing Best Practices

Research compiled from k6, Artillery, Vegeta, oha, Grafana, and industry sources.

---

## What is a Soak Test?

A soak test (endurance test) runs sustained, normal-level load against a system for hours or days to detect problems that only emerge over time: memory leaks, connection pool exhaustion, file handle leaks, latency degradation, and resource accumulation.

**It is NOT a load test.** Load tests find the breaking point. Soak tests find the slow rot.

---

## Metrics That Matter

### The RED Method (Start Here)

From Google SRE's Four Golden Signals, adapted by Grafana:
- **Rate** — requests per second (should remain stable)
- **Errors** — error rate percentage (should stay near zero)
- **Duration** — latency percentiles (should remain flat over time)

### Latency Percentiles

| Metric | What It Reveals |
|--------|----------------|
| p50 (median) | Baseline user experience |
| p95 | The "unlucky 1 in 20" users. Primary SLO target |
| p99 | Tail latency — connection pool contention, GC pauses |
| p99.9 | Worst-case. Only meaningful in long soak tests with millions of requests |

**The defining signal is not absolute latency but latency SLOPE over time.** A p95 that starts at 120ms and ends at 350ms after 8 hours indicates degradation, even if both values are "acceptable" in isolation.

### Error Rate Over Time

Plot error rate per time bucket (e.g., per minute). A soak test may start at 0% errors and creep to 2% after 6 hours. This creep is the signal.

### Status Code Distribution

`{code: count}` histogram. Watch for:
- Growing 429s (rate limiting kicking in)
- Intermittent 502/503 (upstream instability)
- Status `null` or `0` (connection refused, DNS failure, timeout)

### Backend Metrics (Complementary)

These are NOT generated by dripfeed but should be monitored alongside:

| Metric | Symptom |
|--------|---------|
| Heap memory (RSS) | Steady climb = memory leak |
| CPU utilization | Should plateau, not climb |
| Connection pool (idle/waiting) | Growing wait count = pool exhaustion |
| File descriptors | Growing count = handle leak |
| GC pause time | Increasing pressure = memory issues |
| Disk I/O | Log files or WAL growing unbounded |

## Duration Guidelines

| Duration | Use Case |
|----------|----------|
| 10-30 min | CI/CD post-deploy smoke soak |
| 1-4 hours | Quick regression check |
| 8-12 hours | Standard soak test (recommended minimum for production) |
| 24 hours | Comprehensive — catches daily patterns, timezone-triggered issues |
| 48-72 hours | Critical systems — catches GC compaction, weekly patterns |

**Minimum for meaningful results: 4 hours.** Memory leaks growing at a few MB/hour need time to manifest.

## Interval Guidelines

| Interval | Requests/Hour | Use Case |
|----------|---------------|----------|
| 500ms | 7,200 | Aggressive monitoring, high-traffic APIs |
| 1s | 3,600 | Standard soak test |
| 3s | 1,200 | Conservative, quota-friendly |
| 5s | 720 | Minimal footprint |
| 30s | 120 | Heartbeat monitoring |
| 1m | 60 | Lightweight uptime check |

## Anti-Patterns to Avoid

### 1. Soak testing before smoke testing

Running an 8-hour soak when a 30-second smoke test would catch the same bug. Follow the pyramid: smoke -> load -> soak.

### 2. Insufficient duration

A 30-minute "soak test" is a short load test. Memory leaks, connection exhaustion, and resource accumulation need hours to surface.

### 3. Flat, identical requests

Real traffic varies. Mix endpoints, payload sizes, and methods. Use weighted random rotation.

### 4. Only monitoring the client side

Response time and status codes are not enough. Pair with server-side monitoring (Prometheus, Datadog) to catch memory leaks, CPU creep, and pool exhaustion.

### 5. Snapshot analysis instead of trend analysis

Looking at aggregate p95 for the entire run hides the fact that p95 was 100ms in hour 1 and 400ms in hour 8. Always plot percentiles over time.

### 6. No pass/fail criteria

"Looks fine" is not a test result. Define thresholds before the test: `p95 < 500ms`, `error_rate < 1%`. Automate the pass/fail decision.

### 7. Ignoring coordinated omission

If the client waits for a response before sending the next request, slow responses reduce the request rate, hiding the worst latencies. dripfeed avoids this by scheduling requests on a fixed interval regardless of response time.

### 8. Short monitoring retention

Dashboard configured for 1-hour retention during an 8-hour soak. Export reports at end of test. SQLite persists everything.

### 9. Load generator as bottleneck

The machine running the soak test runs out of connections or memory. At dripfeed's low request rates (1-3 req/sec), this is unlikely but worth monitoring for very long runs.

### 10. Not disabling keep-alive (when testing connection management)

Persistent connections hide connection establishment overhead and connection leak bugs. For connection-focused soak tests, set `keepAlive: false`.

## Integration Patterns

### CI/CD (GitHub Actions)

```yaml
name: Post-deploy soak
on:
  deployment:
    types: [completed]

jobs:
  soak:
    runs-on: ubuntu-latest
    timeout-minutes: 20
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
      - run: npx dripfeed run -c soak.config.yaml -d 10m --report json -o results.json -q
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: soak-results
          path: results.json
```

### Scheduled (GitHub Actions cron)

```yaml
on:
  schedule:
    - cron: '0 2 * * *'  # Nightly at 2 AM
```

### Webhook alerting

```yaml
# dripfeed.config.yaml (future v1.1)
alerting:
  webhook: https://hooks.slack.com/services/T.../B.../xxx
  on: threshold_breach
```

### Background job (Inngest / Trigger.dev)

```typescript
import { createSoakTest } from 'dripfeed';

export const nightlySoak = inngest.createFunction(
  { id: 'nightly-soak' },
  { cron: '0 2 * * *' },
  async () => {
    const test = createSoakTest({ /* config */ });
    return await test.run({ duration: '30m' });
  },
);
```

## Report Contents Checklist

A good soak test report includes:

### Essential (MVP)

- [ ] Total requests, success count, failure count, uptime %
- [ ] Duration of test
- [ ] Latency percentiles: p50, p95, p99, min, avg, max
- [ ] Status code histogram
- [ ] Per-endpoint breakdown (requests, avg latency, error count)
- [ ] Error list with sample response bodies
- [ ] Threshold pass/fail results

### Important (v1.1)

- [ ] Latency over time chart (p50, p95, p99 as lines)
- [ ] Error rate over time chart
- [ ] Latency trend detection (slope analysis)
- [ ] Status code distribution over time

### Nice-to-have (v1.2)

- [ ] Run comparison (baseline vs current)
- [ ] Memory/CPU charts (if server metrics are available)
- [ ] Geographic latency breakdown (if multi-region)
